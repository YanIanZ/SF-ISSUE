# Slimefun4 SF1 — SuDo 26.2 + Paper 26.2 Migration

**Date:** 2026-07-10
**Status:** Draft — awaiting user review
**Repo:** `/Users/rheninxy/Client-Project/Slimefun4` (separate from SuDo)
**Target:** Java 25, Paper API 26.2, SuDo `26.2.0`, Folia parity
**Sub-project:** SF1 of the Slimefun4 modernization (SF1 → SF2 → SF3 → SF4)

## Context

Slimefun4 already depends on SuDo (`dev.yanianz:sudo-api:26.1.2`) and has a prior Folia pass
("full Folia support via SuDo Scheduler — ~80 call sites migrated"). But SuDo has since advanced
to `26.2.0` through three merged epics that **broke or moved** several APIs Slimefun4 calls:

- **P2a** deleted the static `Scheduler` (→ instance `SchedulerService` obtained from a `SuDo`
  context), deleted `ChatInput` (→ `ChatService`), and changed `TaskQueue.execute(Plugin)` →
  `execute(SchedulerService)`.
- **P2b** removed `MiniMessages.color()` and made `MiniMessages.alternating(...)` hex-native.

Slimefun4 currently compiles only because mavenLocal still holds the **old** SuDo `26.1.2`.
This slice bumps Slimefun4 onto SuDo `26.2.0` + Paper API `26.2`, migrating every broken call
site so the build is green again, then verifies Folia. It ships **no new gameplay** — that is
SF4. Finishing the `dough → SuDo` migration (119 remaining dough imports) is **SF2**, out of
scope here.

## Goals

1. Publish SuDo `26.2.0` to mavenLocal and bump Slimefun4's `sudo-api` + `paper-api` deps.
2. Give Slimefun4 a single `SuDo` context on its `Slimefun` singleton and route the deleted
   static-API call sites through it.
3. Migrate all P2a/P2b breaking changes: `Scheduler` → `scheduler()`, `ChatInput` → `chat()`,
   `TaskQueue.execute(Plugin)` → `execute(scheduler())`.
4. `./gradlew build` compiles and all existing tests pass against SuDo `26.2.0` + Paper `26.2`.
5. Folia behavior preserved (no reintroduced main-thread assumptions).

## Non-Goals

- New gameplay mechanics (SF4).
- Finishing `dough → SuDo` (SF2) — the 119 remaining `dough` imports stay for now.
- Paper-API-100% / Spigot-legacy audit (SF3).
- Any SuDo library change — SuDo `26.2.0` is consumed as-is.

  `asyncTimer` ×4, `run` ×3, `runLater` ×3, `runTimer` ×2, `cancelAll` ×2. Files include
  `AutoSavingService`, `GitHubTask`, `GitHubService`, `HologramsService`, `IntegrationsManager`,
  `implementation/Slimefun` (main), and others importing `dev.yanianz.sudo.scheduling.Scheduler`.

  `ChatInput.waitForPlayer(Slimefun.instance(), p, callback)` — in `ChatUtils`,
  `SurvivalSlimefunGuide`, `ProgrammableAndroid`, `GPSNetwork`.
- **`TaskQueue.execute(Plugin)`** (→ `execute(SchedulerService)`): calls
  `queue.execute(Slimefun.instance())` in `AutomatedPanningMachine`, `MiningTask` (×2),
  `Composter`. (The `instruction.execute(...)` / android `execute(...)` hits are unrelated
  methods — do not touch.)
- **NON-issues (verified, no change needed):**
  - `MiniMessages.color()` — 0 usages in Slimefun4.
  - `MiniMessages.alternating(...)` — Slimefun4's `ChatColors.alternating(String, ChatColor...)`

    calls `MiniMessages.alternating`, so P2b's signature change does not affect it.
  - `UUIDLookup.getUuidFromUsername(Plugin, String)` and `PlayerHead.getItemStack`/`setSkin`

## Design

### 1. Dependency + platform bump

- **Prerequisite:** publish SuDo `26.2.0` to mavenLocal so Slimefun4 can resolve it:
  `cd ../SuDo && ./gradlew publishToMavenLocal` (Slimefun4's `build.gradle.kts` already lists
  `mavenLocal()` first). Verify `~/.m2/repository/dev/yanianz/sudo-api/26.2.0/` exists.
- In Slimefun4 `build.gradle.kts`:
  - `implementation("dev.yanianz:sudo-api:26.1.2")` → `26.2.0`.
  - `compileOnly("io.papermc.paper:paper-api:26.1.2.build.72-stable")` → the exact 26.2
    coordinate SuDo pins (`26.2.build.56-alpha`; re-verify the newest 26.2 build at impl the same
    way SuDo did — Paper API is `compileOnly`, so alpha is safe).
  - The shadowJar `relocate("dev.yanianz.sudo", "...libraries.sudo")` stays unchanged.

### 2. `SuDo` context on the `Slimefun` singleton

`Slimefun` (the JavaPlugin singleton, `implementation/Slimefun.java`) becomes the owner of one
`SuDo` context for the whole plugin:

- Add `private SuDo sudo;` (import `dev.yanianz.sudo.SuDo`).
- Early in `onEnable()` (before any service that schedules), `this.sudo = SuDo.create(this);`.
- Accessors:
  - `public SuDo sudo()` — the context.
  - `public SchedulerService scheduler()` → `sudo.scheduler()`.
  - `public ChatService chat()` → `sudo.chat()`.
  - `public SkinService skins()` → `sudo.skins()` (added for symmetry / SF2 use; only wire it if a
    call site needs it this slice).
- In `onDisable()`, replace `Scheduler.cancelAll(this)` with `sudo.shutdown()` (which cancels
  scheduler tasks and disposes the chat listener). Remove the `dev.yanianz.sudo.scheduling.Scheduler`
  import from `Slimefun.java`.

The existing `Slimefun.instance()` static accessor is the reach path for call sites:
`Slimefun.instance().scheduler()...`.

### 3. Call-site migrations (mechanical)

- **Scheduler → scheduler()** (drop the `Plugin`/`Slimefun.instance()` first arg):
  - `Scheduler.run(plugin, t)` → `Slimefun.instance().scheduler().run(t)`
  - `Scheduler.runLater(plugin, t, d)` → `...scheduler().runLater(t, d)`
  - `Scheduler.runTimer(plugin, c, d, p)` → `...scheduler().runTimer(c, d, p)`
  - `Scheduler.async(plugin, t)` → `...scheduler().async(t)` (if present)
  - `Scheduler.asyncLater(plugin, t, d)` → `...scheduler().asyncLater(t, d)`
  - `Scheduler.asyncTimer(plugin, t, d, p)` → `...scheduler().asyncTimer(t, d, p)`
  - `Scheduler.cancelAll(this/plugin)` → `Slimefun.instance().scheduler().cancelAll()` (or
    `sudo.shutdown()` in `onDisable` per §2).
  - Remove the now-unused `import dev.yanianz.sudo.scheduling.Scheduler;` from each of the 11 files.
  - Where a class already holds a `Slimefun`/plugin reference, use it; otherwise
    `Slimefun.instance()`.
- **ChatInput → chat()** (4 sites):
  `ChatInput.waitForPlayer(Slimefun.instance(), p, cb)` →
  `Slimefun.instance().chat().waitForPlayer(p, cb)`. Remove the `ChatInput` import.
  (`ChatService.waitForPlayer` mirrors the old overloads — `Consumer<String>`,
  `Predicate + Consumer`, `BiConsumer`, `Function<String, ChatResult>` — minus the leading
  `Plugin` arg.)
- **TaskQueue.execute** (~4 sites):
  `queue.execute(Slimefun.instance())` → `queue.execute(Slimefun.instance().scheduler())`.
- **Import verification:** after migrating, confirm every remaining `dev.yanianz.sudo.*` import in
  Slimefun4 resolves against `26.2.0` (the untouched SuDo modules — `protection.Interaction`,
  `data.persistent.PersistentDataAPI`, `blocks.BlockPosition`, `inventory.InvUtils`,
  `collections.*`, `versions.*`, `updater.*`, `recipes.MinecraftRecipe`, `common.*`,
  `items.ItemUtils`, `data.TriStateOptional`). These modules were not changed by P1/P2a/P2b, so
  they should compile unchanged; the full build is the gate.

### 4. Folia + verification

- `./gradlew build` — must compile and pass all existing tests (MockBukkit `mockbukkit-v1.21`).
- Audit: `grep -rn "dev.yanianz.sudo.scheduling.Scheduler\b"` over `src/main` returns nothing (the
  static facade is gone from Slimefun4 too). No direct `Bukkit.getScheduler()` /
  `BukkitRunnable` reintroduced.
- Folia: the scheduling now flows through `SchedulerService` (which wraps SuDo's Folia-aware
  backend), so the prior Folia migration is preserved. Document a manual Folia 26.2 smoke test
  (boot Folia, load Slimefun, verify a machine ticks + auto-save runs without wrong-region-thread
  errors) — automated tests run on Paper via MockBukkit.

## Success Criteria

1. mavenLocal has SuDo `26.2.0`; Slimefun4 `build.gradle.kts` pins `sudo-api:26.2.0` +
   `paper-api:26.2...`.
2. Zero references to the deleted `dev.yanianz.sudo.scheduling.Scheduler` and
   `dev.yanianz.sudo.chat.ChatInput` remain in `src/main` (`grep` clean).
3. `Slimefun` owns one `SuDo` context; `onDisable` calls `sudo.shutdown()`.
4. `./gradlew build` is green — compiles against SuDo `26.2.0` + Paper `26.2`, all tests pass.
5. No reintroduced main-thread scheduling; Folia smoke test documented.

## Risks

- **Paper 26.2 is pre-release (alpha).** `compileOnly` contains the blast radius; if a removed
  Paper API blocks compilation, apply the minimal documented replacement (same posture as SuDo P1)
  or, if the user prefers, stay on Paper `26.1.2` stable for SF4 and bump when 26.2 ships.
- **SuDo `26.2.0` must be published locally first** — if `publishToMavenLocal` is skipped, the
  dep won't resolve. Task 1 does it explicitly.
- **Untouched-module import drift:** a SuDo module SF4 imports could have changed a package/name
  between the mavenLocal `26.1.2` and the current `26.2.0` source. Mitigation: the full
  `./gradlew build` is the gate; any unresolved import is fixed by pointing at the real `26.2.0`
  symbol (verify against SuDo source, do not invent).
- **Scheduler `run`-vs-`runTimer` signature:** SuDo `SchedulerService.runTimer` takes a
  `Consumer<Integer>`; confirm each migrated timer call passes a `Consumer<Integer>` (the old
  static `Scheduler.runTimer` had the same shape) — adjust lambdas if a call passed a bare
  `Runnable`.
